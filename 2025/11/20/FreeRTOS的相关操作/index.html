对于初学者来说，内存管理确实是嵌入式开发中最容易让人头疼、也最核心的概念。我们先从硬件基础聊起，再到如何为任务“量体裁衣”。

一、 核心概念扫盲：RAM, SRAM 到底是什么？

你可以把单片机想象成一间办公室：

  • Flash (硬盘/仓库)：存放你编写的代码程序。断电后数据不会丢失。
  • RAM (桌面/工作区)：程序运行时存放临时变量、函数调用栈的地方。断电后数据立刻清零。
    • SRAM (Static RAM)：是 RAM 的一种，STM32 内部集成的就是这种。它的特点是速度极快，不需要像电脑内存（DRAM）那样频繁刷新，非常适合嵌入式。
  • 栈 (Stack)：它是 RAM 空间里的一块特殊区域，遵循“后进先出”。每当你调用一个函数、定义一个局部变量，都会占用这里的空间。

你的芯片 STM32F103C8T6 有多少空间？

  • Flash: 64 KB （能存 64,000 字节的代码）。
  • SRAM (RAM): 20 KB （这就是你的“总桌面大小”）。

注意：在 RTOS 中，你所有的任务堆栈、全局变量、内核开销，都必须挤在这 20 KB 里面。

二、 怎么确定分配给任务多大的栈？

在 RTOS 中，任务堆栈的大小（usStackDepth）单位通常是 Word (字)。对于 STM32（32位机），1 Word = 4 字节。

1. 任务栈里到底装了什么？

分配栈空间时，你需要考虑以下三部分的总和：

  1. 局部变量：例如 char buf[50]; 直接占用 50 字节。
  2. 函数调用链：A函数调用B函数，B调用C，每一层都需要保存“现场”（返回地址和寄存器），嵌套越深占用越多。
  3. 中断上下文：当任务被切换时，RTOS 会把当前 CPU 的所有寄存器（R0-R15 等）全部压入该任务的栈中保存。这部分固定开销大约在 64~128 字节 左右。

C

  • 估算：固定开销(128) + 变量(24) + 函数深层调用(200) ≈ 352 字节。
  • 安全余量：我们通常给 1.5 到 2 倍的余量。
  • 分配值：352 * 2 = 704 字节。换算成 Word：$704 / 4 = 176$。
  • 经验法则：对于 STM32，简单任务给 128 Word (512B)，复杂任务（带 printf 或大量运算）给 256 Word (1KB)。

三、 实战：怎么知道栈给多了还是给少了？

既然“估算”不准，RTOS 提供了专门的工具来监控。

1. 静态观察：编译输出

看你第一张图的 Build Output：

  • RW-data: 已初始化的全局变量。
  • ZI-data: 未初始化的全局变量（包含你为任务定义的 CPU_STK TASK_STACK[SIZE] 数组）。

公式：RW-data + ZI-data 必须小于芯片的 RAM 总量（20KB）。如果这个值接近 20000，程序就很危险了。

2. 动态监测：高阶技巧

uC/OS-III 有一个非常实用的功能叫 堆栈检测 (OS_OPT_TASK_STK_CHK)。

  • 你可以调用函数获取某个任务的“历史最高栈使用量”。
  • 原则：如果一个任务分配了 256 字，运行一天后最高只用了 80 字，那你就可以把空间缩减到 128 字，节省内存。

四、 给初学者的建议步骤

  1. 初期“大方一点”：C8T6 虽然只有 20KB，但跑 4-5 个简单任务没问题。每个任务统一先给 128 或 256 (Word)。
  2. 警惕局部大数组：不要在任务里定义 int sensor_data[100]; 这种局部数组，这会直接撑爆任务栈。改用 static 或全局变量。
  3. 关注报错：如果程序莫名其妙卡死在 HardFault_Handler， 90% 的概率是某个任务栈溢出了。
